## 捨てた選択肢と理由

- `tmux/SSH` を日中運用の主経路にする案:
  - スマホからの表示崩れ、スクロール、長文入力が厳しく、承認コンソールとして不向き。
- Discotice を単なる command bridge のまま使う案:
  - 承認待ち、永続化、run state、監査が足りず、長時間の半自動開発に耐えない。
- 危険コマンド承認を第1段階で同時実装する案:
  - `merge / hold / retry` より価値が低く、初期実装を重くするため後回しにした。
- AgentTeam を runner の代わりにして Issue 間オーケストレーションまで持たせる案:
  - Issue 間の dispatch、再試行、承認待ち管理は別の runner 層で持つ方が責務分離しやすい。

## 学び

- Symphony は「1 Issue 内の多エージェント協調」より、「Issue 間の dispatch / retry / approval 待ち / cleanup」を担うランナー層として捉えると整理しやすい。
- Claude Code Agent Teams は `subagents` とは別物で、独立セッション群を leader が束ねる構造として見るのが正確。
- ベンダー非依存にしたいなら、`leader/worker` 機能そのものより、runner 主導の `task bus + run state + approval state` に抽象化する方が移植しやすい。
- Discotice の次の価値は「Discord 入力ブリッジ」ではなく、「通知・承認・状態同期の運用層」への拡張にある。
- Codex のグローバル既定指示は `/home/az/.codex/config.toml` の `developer_instructions` に入れられる。
